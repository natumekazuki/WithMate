# コードレビュー記録 — workspace-file-search-index

- **観点**: workspace 外 ignore ソースの追従、ignore ファイル状態取得の整合性

## 指摘一覧（2件）

### P2 新規作成された外部 ignore ファイルも失効条件に含める

- **位置**: `src-electron/workspace-file-search.ts:122-129`
- **問題**: 前回の scan 時に `.git/info/exclude` や workspace 外の親 `.gitignore` が存在しなかった場合、`index.ignoreFiles` にそのパスが入らないため、このループでは新しい ignore ソースの出現を検知できない。これらのディレクトリは `visitedDirectories` にも含まれないため、TTL 超過後でもキャッシュが延命され続ける。
- **影響**: `@path` 検索結果が新しい ignore ルールを反映しないまま残り、workspace 外で追加された ignore 設定が永続的に無視される。
- **対応方針**: 少なくとも既知の外部 ignore 配置候補については、「存在しなかったこと」も含めて状態として保持し、次回検証時に新規出現を検知できるようにする。

### P3 ignore ファイルの内容と mtime を同じ版で取得する

- **位置**: `src-electron/snapshot-ignore.ts:121-126`
- **問題**: `readFile()` と `stat()` を並列実行すると、scan 中に `.gitignore` が編集された場合に「古いルール内容」と「新しい mtime」を組み合わせてキャッシュする競合が起きうる。
- **影響**: `checkStructureUnchanged()` は新しい mtime を見て一致判定してしまうため、古い ignore ルールのまま `@path` 検索結果が残り、別のディレクトリ変更が起きるまで自己修復しない可能性がある。
- **対応方針**: `stat()` と内容取得を逐次化するか、読み取り前後で `mtime` を再確認して、整合した版だけをキャッシュに採用する。

---

## 要約

- workspace 外の ignore ソースは、新規作成時も含めて失効条件に入れないとキャッシュが無期限に古いまま残る。
- ignore ファイルの内容とメタデータは同じ版で取得しないと、stale な matcher を固定化する競合が残る。
