- **対象**: `src-electron/workspace-file-search.ts`
- **観点**: `@path` 検索インデックスのキャッシュ整合性、stale 判定、ignore ルール追従

## 指摘一覧（2件）

### P1 `.gitignore` 変更時にもキャッシュを失効させる

- **位置**: `src-electron/workspace-file-search.ts:98-104`
- **問題**: `checkStructureUnchanged()` は `visitedDirectories` の mtime しか見ていないため、既存の `.gitignore` / 親ディレクトリの `.gitignore` / `.git/info/exclude` を編集しても TTL 更新後にずっと `unchanged` 扱いになる。
- **影響**: `scanWorkspacePaths()` の結果は ignore ルールに依存するため、無視対象から外したファイルが `@path` 検索に永続的に出てこない、あるいは新しく無視したファイルが残り続ける回帰になる。
- **対応方針**: ディレクトリ mtime だけでなく、走査時に参照した ignore 関連ファイル群の状態もキャッシュ検証対象に含める。

### P2 走査完了後の時刻で `validatedAt` を記録する

- **位置**: `src-electron/workspace-file-search.ts:121`
- **問題**: ここで取得した `now` が後続の `scanWorkspacePaths()` / `checkStructureUnchanged()` 完了後にもそのまま `validatedAt` に使われるため、大規模 workspace や遅いストレージでこの非同期処理が TTL を超えると、新しい index が保存された直後から stale 扱いになる。
- **影響**: 次の `@path` 検索が即座に再検証や再走査を始め、意図したキャッシュ最適化が効かなくなる。
- **対応方針**: `validatedAt` は検証または走査の完了直後の時刻で更新する。

---

## 要約

- ignore ルール変更をキャッシュ失効条件に含めないと、`@path` 検索結果が永続的に古いまま残る。
- `validatedAt` を処理開始時刻で保存すると、長い走査の直後に index が即 stale になる。
